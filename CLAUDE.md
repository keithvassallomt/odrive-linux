# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

odrive-linux is a native Linux frontend for [odrive](https://www.odrive.com)'s on-demand cloud sync. odrive ships an official headless agent (`odriveagent`) and a CLI (`odrive`) that operate on **placeholders** — zero-byte stand-ins for remote files/folders that materialize on demand:

- `*.cloud` — placeholder for a single remote file
- `*.cloudf` — placeholder for a remote folder

This repository builds a higher-level manager around that agent: a Rust CLI, a GTK4/Libadwaita GUI, and a Python Nautilus extension. **None of this code talks to odrive's cloud directly** — every sync/unsync/status call shells out to the user's installed `odrive` binary at `~/.odrive-agent/bin/odrive`.

**Design intent** — the manager wraps the agent so a GNOME user gets:
1. **Daemon orchestration** — make sure `odriveagent` is up (systemd-user preferred, `nohup` fallback) and surface authenticate/mount as part of onboarding.
2. **Local state for the UI** — scan the sync tree for `.cloud`/`.cloudf` and persist them in SQLite so the GUI doesn't have to re-shell-out for every paint.
3. **Transparent on-demand sync** — Nautilus right-click triggers `odrive sync` against a `.cloud` placeholder so the file materializes before the OS hands control back.
4. **Space management** — right-click "Unsync" calls `odrive unsync`, with the agent's `autounsyncthreshold` available as a longer-term auto-cleanup knob.

## Build / run

The repo is a Cargo workspace with three crates plus one standalone Python file.

```bash
cargo build                          # build all three crates
cargo build --release                # release build
cargo check                          # fast type-check across the workspace
cargo run -p odrive-cli -- <subcmd>  # run the CLI (status, mounts, sync, unsync, refresh, scan, start, stop)
cargo run -p odrive-gui              # launch the GTK app (wizard or dashboard depending on state)
cargo test --workspace               # 26 unit tests across odrive-core (config, db, parsers, threshold round-trip, folder-rule CRUD)
```

The GUI requires GTK4 and Libadwaita 1.5+ system libraries (`libgtk-4-dev`, `libadwaita-1-dev` on Debian/Ubuntu).

## Runtime prerequisites the code assumes

These are not installed by this repo — the user must have them set up before any of this code is useful:

1. **odrive agent binaries** — `odrive` and `odriveagent` in a directory `OdriveAgent::new()` reads from `~/.config/odrive-linux/config.toml` (key: `agent_bin_dir`, default `~/.odrive-agent/bin`). The GUI's onboarding wizard installs them on first launch via the official `dl.odrive.com` pipeline if missing, or accepts a custom path.
2. **Authentication** — `odrive authenticate <key>`, surfaced through the wizard's Login page or invokable directly.
3. **At least one mount** — `odrive mount <local> <remote>`. The wizard offers an optional Mount page (`~/odrive` ↔ `/`) at the end; users can skip and mount later.
4. **State DB** at `~/.odrive-linux.db` — created automatically on first `scan` or GUI launch.
5. **`python3-nautilus`** (only for the right-click integration) — Debian/Ubuntu `python3-nautilus`, Fedora `nautilus-python`, Arch `python-nautilus`. Without it the extension file is never loaded by Nautilus, even when symlinked into `~/.local/share/nautilus-python/extensions/`. Not auto-installed by anything in this repo.
6. **`gnome-shell-extension-appindicator`** (only for the panel indicator on stock GNOME) — Ubuntu/Pop!_OS/Fedora Workstation install it by default; Arch, Debian, vanilla GNOME do not. Without it the StatusNotifierItem-based tray icon registers on D-Bus but never gets rendered by GNOME Shell. KDE/XFCE/Cinnamon render it natively. The Manager logs a single warning (`Tray indicator unavailable …`) and continues working without the icon if the extension isn't present.
7. **`wl-clipboard` / `xclip`** — *only* needed as a fallback for the Nautilus right-click "Copy Share Link" action; the primary path uses GTK4's in-process `Gdk.Clipboard.set_content` so no extra package is required on a normal GNOME setup. The "Share Link" (open in browser) action uses `xdg-open` and has no extra prereq.

The first three are walked through automatically by the GUI's onboarding wizard on first launch. See "Onboarding wizard" below.

## Architecture

```
odrive-cli  ──┐                 ┌─► <agent_bin_dir>/odrive   (configured via)
              ├──► odrive-core ─┼─► ~/.odrive-linux.db        (SQLite — placeholder rows)
odrive-gui  ──┘                 └─► ~/.config/odrive-linux/config.toml  (XDG — agent_bin_dir)

nautilus_extension.py  ──► odrive-cli (via _find_cli) ──► (same path through odrive-core)
```

**`odrive-core`** is the only crate that knows how to talk to the agent. Three types matter:

- `OdriveAgent` (lib.rs) — wraps every CLI subcommand the manager uses (`status`, `status --mounts`, `sync`, `unsync`, `refresh`, `mount`, `authenticate`) plus daemon lifecycle (`start`/`stop` try `systemctl --user start odrive.service` first, fall back to `nohup odriveagent`). Onboarding helpers: `is_authenticated()`, `install_official()` (runs the dl.odrive.com curl+tar pipeline), `write_systemd_unit()` (templates the unit's `ExecStart` with the current `agent_path`), `enable_systemd_unit()`, `enable_linger()`. Path-resolution helpers: `default_mount_path()` (`~/odrive`), `with_new_bin_dir()` (rebuild after the wizard's "specify custom location" branch). Also contains `scan_placeholders`, which walks a mount tree and upserts every `.cloud`/`.cloudf` it finds into the DB.
- `OdriveDb` (db.rs) — thin rusqlite wrapper around three tables. `placeholders` (id, local_path, remote_path, is_folder, sync_status) bridges "what the CLI tells us" → "what the GUI renders" — the GUI never invokes the agent for placeholder counts, it reads from SQLite. `folder_sync_rules` (id, local_path UNIQUE, threshold_mb, expand_subfolders, created_at, paused_threshold_mb) is the GUI's source of truth for "did the user set a foldersyncrule via the Manager?", because the upstream agent has no LIST or REMOVE for its own foldersyncrule storage. `threshold_mb` uses sentinel values: `0` = never, `-1` = unlimited (`inf`), positive integer = MB; `FolderSyncThreshold::{to,from}_db_value` straddle that encoding. `paused_threshold_mb` (nullable, added in a forward-only migration in `OdriveDb::migrate`) holds the prior threshold while a rule is paused — `pause_folder_rule(path)` stashes the value there and zeroes the live `threshold_mb`; `resume_folder_rule(path)` restores. The agent has no native pause concept, so this approximates: paused = "0 = never download" upstream + Manager-side memory of what to restore. `FolderRule::is_paused()` is the canonical render predicate. `sync_in_progress` (id, local_path UNIQUE, started_at) is cross-process state for the syncing emblem: the GUI marks before kicking off `sync_recursive` and clears on completion (see `mount_detail.rs::spawn_sync`); the Python Nautilus extension reads it (with a 0.5s TTL cache) on every `update_file_info`. The dashboard sweeps stale rows at startup so a GUI crash mid-sync doesn't leave a permanent emblem.
- `OdriveConfig` (config.rs) — `~/.config/odrive-linux/config.toml`. Currently a single key, `agent_bin_dir`, set by the wizard's "specify custom location" branch. Load is fault-tolerant: missing file or unparseable TOML both fall back to defaults rather than erroring (a fresh-system run is the common case, not a bug).

**`odrive-cli`** is a clap-derive front-end that 1:1 maps subcommands onto `OdriveAgent` methods. `Status` and the no-subcommand default both print agent status plus the DB-tracked placeholder count.

**`odrive-gui`** is a Libadwaita app split into two surfaces, each its own `ApplicationWindow`:

- **Onboarding wizard** (`wizard.rs`) — `Adw.NavigationView` with up to four pages (Install / Service / Login / optional Mount). At `connect_activate`, `main.rs::needs_wizard()` checks all four preconditions; if any fails the wizard window is presented, otherwise the dashboard goes straight up. Pages advance dynamically: each successful action re-runs `push_next` which checks every precondition fresh and pushes the next failing one (or closes the wizard). The wizard's agent is held in `Rc<RefCell<OdriveAgent>>` because the Install page can swap the active bin directory mid-flow. The install download (curl + tar from dl.odrive.com) runs on a worker via `worker::spawn`; mount and authenticate are quick CLI calls and stay on the main thread.
- **Dashboard** (`main.rs::present_dashboard`) — wrapped in an `Adw.NavigationView` so mount-detail pages can be pushed onto the same window. The shell is a four-tab `Adw.ViewStack` driven by an `Adw.ViewSwitcher` (placed as the headerbar's title widget) with `policy = Narrow` to render icon-only tabs (per the spec — no text labels in the bar). Tabs: **Mount & Sync** (`folder-symbolic`, mounts list + sync-rules list + folder navigation, built by `build_mount_sync_page`), **Backup** (`drive-harddisk-symbolic`), **Encrypt** (`channel-secure-symbolic`), **Trash** (`user-trash-symbolic`). The latter three are stub `StatusPage`s today, materialised by `stub_status_page`, and will get real content as those features land. Each mount row gets a trailing "Unmount" button that pops an `Adw.MessageDialog` confirmation before calling `agent.unmount(local)`. The header keeps the hamburger menu (`Preferences`, `About`). The Agent panel (status + start/stop) and Placeholder DB panel that used to sit above the mounts list both relocated to **Preferences → Status** when the dashboard became a focused-on-mounts surface. The Mount & Sync tab's 5s `glib::timeout_add_seconds_local` poll rebuilds the mounts and sync-rules groups each tick; ticks fire whether or not the tab is the visible child, since the widget tree exists either way.

  The **Sync Rules** group lists every row in `OdriveDb::list_folder_rules()` with a paused/active prefix icon and a subtitle that renders the threshold (`format_threshold` / `format_rule_subtitle` — paused rules show "Paused — would auto-download X" so the would-resume value is visible). Body click → `mount_detail::build_folder_page` pushes the corresponding folder's detail page onto the shared NavigationView (same one the mount drill-in uses), so the rule's editor is one click away. Trailing ellipsis → `Gtk.Popover` with three flat buttons:
    - **Open folder** — same as body click.
    - **Pause** / **Resume** — calls `toggle_pause`, which DB-mutates first (`pause_folder_rule` / `resume_folder_rule`) then pushes to the upstream agent (threshold `0` for pause, the restored threshold + saved `expand_subfolders` for resume). DB-first ordering means the next tick reads the right paused state even if the agent push fails.
    - **Delete** (destructive-action styled) — `delete_rule` pushes threshold `0` upstream (since there's no remove command) then drops the row from `folder_sync_rules`.
- **Preferences window** (`settings_page.rs`) — opens as a standalone modeless `ApplicationWindow` (not a page pushed onto the dashboard's NavigationView) shaped like GNOME Settings: an `Adw.NavigationSplitView` with a `Gtk.ListBox` sidebar on the left and a `Gtk.Stack` of section pages on the right. Sections (in `SECTIONS` order): General, Appearance, Advanced, Status. General is a `PreferencesPage` with `Adw.ComboRow` widgets for `PlaceholderThreshold`, `XlThreshold`, and `AutoUnsyncThreshold` — the earlier General/Premium split predated odrive removing the free tier; with no account-state distinction left, all three thresholds live in General together. Appearance hosts two groups: "Panel indicator" (tray-icon-colour combo) and "Nautilus emblems" (two `Adw.SwitchRow`s: synced + syncing — both default on, persisted to `OdriveConfig`'s `nautilus_synced_emblem` / `nautilus_syncing_emblem`; the Python extension reads them on each `update_file_info` via a 1 s TTL cache so toggling takes effect within a directory tour and survives a Nautilus restart). Status is a `PreferencesPage` that hosts the Agent panel (status row + start/stop button), Placeholder DB row (count + Scan-now button), and a Logs row with **View** / **Open** buttons (View opens `log_viewer::present` for a live colorized tail of `~/.odrive-agent/log/main.log`; Open shells `xdg-open` on the log directory). Status runs its own 5 s `glib::timeout_add_seconds_local` poll cancelled in `connect_close_request` to avoid a leaked timer holding clones of the now-unrooted widgets. Advanced is a placeholder `StatusPage` today. Selection changes apply immediately (no Save button — same idiom as GNOME Settings) by calling the matching `OdriveAgent` setter (or, for the tray colour, `TrayController::set_icon_color` which re-renders the SNI pixmap on the running indicator). On any CLI failure the row is reverted to the value the agent reports, gated by a shared re-entrancy `RefCell<bool>` to keep the revert from re-firing the handler. A single `ToastOverlay` wraps the Stack so toasts surface on whichever section is active.
- **Panel indicator** (`indicator.rs`) — StatusNotifierItem (the maintained successor to AppIndicator) registered via the `ksni` crate's blocking API. Lives inside the GUI process; the icon disappears when the GUI quits. The tray runs on its own background thread; menu callbacks post `TrayEvent`s on a `std::sync::mpsc` channel that a `glib::timeout_add_local` 150ms poll on the GTK main loop drains, so every GUI-touching action (window present, xdg-open, agent start/stop/shutdown) runs from GTK. Icon colour is configurable via `OdriveConfig::tray_icon_color` (default `pink`; see `TRAY_ICON_COLORS` for the valid set) and resolved against `odrive-tray-<color>` icons installed by `odrive-cli install-handlers`. `install()` returns a `TrayController` the Settings page holds onto so the colour combo can swap the icon live without restarting. Menu items: Open odrive folder (`xdg-open` on first mount's local path), Open odrive Manager (`window.present()`), Pause/Resume sync (toggles `agent.stop()` / `agent.start()` via `worker::spawn`; same heavy-handed approximation that the Settings page uses, since upstream has no pause command), Quit (just `app.quit()` — the agent is independent infrastructure managed by systemd and must outlive the GUI; killing it via `odrive shutdown` would stop sync until next login). A 2s `glib::timeout_add_seconds_local` poll calls `agent.get_sync_activity()` and mirrors `is_running` into the tray title + Pause/Resume label. The same poll drives the **active-state animation**: when `SyncActivity::is_active()` flips true (any of `Sync Requests`, `Background Requests`, `Uploads`, `Downloads`, `Waiting` > 0), we start an 80ms animation timer that cycles through the 16 frames installed as `odrive-tray-<color>-active-<N>` (see "Tray animation frames" non-obvious below) by pushing each frame's pre-rendered pixmap via `Handle::update`; on transition back to idle we `.remove()` the timer's `SourceId` and push the static idle pixmap. Pre-rendered frames live in `Rc<RefCell<RenderedSet>>` shared between the activity poll, the animation tick, and `TrayController::set_icon_color` (which rebuilds the set on colour change). Colours without their own bundled animation set (`darkgrey`, `grey` today) fall back to the default colour's frames (pink) — `RenderedSet::for_color` tries the chosen colour first, then `DEFAULT_TRAY_ICON_COLOR` if that returns nothing. The idle icon stays in the user's chosen colour either way, so the panel still reflects their appearance pick when nothing is in flight. If `ksni::blocking::TrayMethods::spawn` fails (no SNI host on the bus, e.g. stock GNOME without the appindicator extension) we log a one-shot `eprintln!` and the rest of the Manager works as before.
- **Log viewer** (`log_viewer.rs`) — standalone `ApplicationWindow` containing a `Gtk.TextView` that live-tails `~/.odrive-agent/log/main.log`. Opens with the existing file's contents pre-loaded so the user lands on recent context, then a 250 ms `glib::timeout_add_local` poll seeks to the saved byte offset and appends new bytes. Lines are tagged by their level token (third whitespace-separated field of the `DD Mmm HH:MM:SSAM LEVEL message` format the agent emits) — `WARN/WARNING` → orange `#c25d00`, `ERROR` → bold red `#c01c28`, anything else (`INFO`, unknown) → default. Auto-scroll to the bottom only when the user is already at the bottom (heuristic on the `ScrolledWindow`'s vadjustment with a 2 px tolerance), so reading older context doesn't get yanked away by new lines arriving. Log rotation is detected by inode change: the file handle is dropped and reopened from byte 0 on the next tick, but the buffer is not cleared so the tail of the prior file remains visible. The poll's `SourceId` is removed in `connect_close_request` to avoid leaking the timer (and therefore the widget clones the closure holds) after the window closes.
- **Mount detail / folder pages** (`mount_detail.rs`) — clicking a mount row on the dashboard pushes `build_mount_root`; the **Sync Rules** list on the Mount & Sync tab pushes `build_folder_page` (same renderer, `is_mount_root=false`). That module's private `build_page` recurses for every drill-in: each subfolder click pushes another page on the same `NavigationView`. The mount root only offers a "First-time setup" StatusPage with an "Expand placeholders" button when the heuristic `appears_expanded` says no real subdirectories exist yet; it then runs `agent.sync_recursive(path, no_download=true)` and re-renders. Subfolder pages show a "Sync rule" `PreferencesGroup` with an Operation combo (Automatic/One-Time), a Download Threshold combo (None/Small/Medium/Large/All/Custom MB), an Apply-to-subfolders switch, a Pause/Resume row (when a rule exists), and a Save row that toggles to Update / Update & resume + a destructive Delete after a rule exists. When a rule is paused, the group title becomes "Sync rule (paused)" and the threshold + expand widgets render the would-resume values (`paused_threshold_mb`) rather than the suspended `0`. Save calls `agent.folder_sync_rule` and persists to `OdriveDb::upsert_folder_rule`; the upsert always sets `paused_threshold_mb = NULL`, so saving a paused rule implicitly resumes it (matching the "Update & resume" button label). Pause/Resume calls the shared `crate::toggle_pause` helper from main.rs (DB mutation first, then upstream push). Delete pushes `FolderSyncThreshold::None` upstream (since there's no remove command) and drops the local DB row. Each page connects `NavigationPage::shown` to re-render on back-nav, so a parent page's badges and rule editor stay in sync after a child page mutates state.

**`nautilus_extension.py`** plugs into Nautilus's `MenuProvider` (right-click) and `InfoProvider` (per-file decoration). On right-click it inspects selected files and — if any are inside a known mount — surfaces a single "Odrive ▸" parent item (icon `odrive-menu`, installed under `apps/` by `install-handlers` from `odrive-icons/app-icon/`) whose submenu contains: **Sync** (if the selection has placeholders), **Unsync** (if it has materialised files), **Refresh** (always, run against every in-mount path), **Share Storage** (static `xdg-open` of `https://www.odrive.com/features/spaces` — odrive Spaces is account-level, not per-item, so the action ignores the selection), **Copy Share Link** (calls `odrive-cli sharelink <path>` per item, newline-joins the URLs, and writes them to the system clipboard via `Gdk.Clipboard.set_content` — we're already inside Nautilus's GTK4 process so no `wl-clipboard`/`xclip` install is needed; those external tools remain as a fallback if `Gdk.Display.get_default()` returns None for any reason), and **Open Web Preview** (calls `odrive-cli weburl <path>` to compose `https://www.odrive.com/browse/<remote-path>` from the mount table and `xdg-open`s the result — pure URL construction, no upstream CLI command for it). Items are emitted only when meaningful for the selection — no greyed-out entries. Each `sharelink` invocation generates a *new* URL (no upstream "fetch existing" form), so re-invoking Copy Share Link on the same selection produces different IDs each time. macOS/Windows ship a richer **Share Link** item that opens odrive's web sharing dialog (configure permissions / expiry / password before generating); that flow needs the agent's session token from `~/.odrive-agent/db/odrive.db`, which is encrypted with SEE (Hipp/Wyrick's commercial codec, AES-256). SQLCipher and other open-source SQLite encryption tools are binary-incompatible with SEE — confirmed by symbol inspection of the bundled `apsw.so` (`sqliteCodecAes256`, the SEE copyright banner). The same wall blocks an "Open in Google Docs / OneNote" item for `.gdoc.cloud` etc. via `/rest/link/get_open_url`. Both deferred indefinitely. `update_file_info` paints emblems based on a precedence: `odrive-syncing` (highest) if the path with any `.cloud(f)` suffix stripped is in `sync_in_progress`; else `odrive-synced` on regular files inside a mount (not the mount root) and on directories that have a row in `folder_sync_rules` (a sync rule promises the folder will be kept in sync — emblem is honest); else nothing — directories without a rule don't get the synced emblem even if their `.cloudf` was expanded, because their contents may still be placeholders and badging the wrapper would be misleading. Each emblem is independently gated by a user-toggleable preference (`nautilus_synced_emblem` / `nautilus_syncing_emblem` in `~/.config/odrive-linux/config.toml`, defaulting to true) read via a tiny line-based TOML scanner with a 1 s TTL cache so flipping the switch in Preferences takes effect within a directory tour. Both DB tables are read with a single 0.5s TTL cache; the GUI also touches the folder's mtime on sync mark/clear, on rule save, and on rule delete, so Nautilus's directory-listing inotify fires and the emblem flip is near-instant rather than waiting on the cache TTL. The extension also opportunistically pads zero-byte placeholders with a single null byte (see "Double-click intercept" below). Both shell out to `odrive-cli`, located via `_find_cli`: `$ODRIVE_CLI` override → `$PATH` lookup → `target/release/odrive-cli` → `target/debug/odrive-cli` (relative to the extension file). If none resolve, the extension loads but stays inert and prints a one-shot stderr hint at init. The mount list is discovered at extension init via `odrive-cli mounts --paths` and cached for the lifetime of the Nautilus process — restart Nautilus (`nautilus -q`) to pick up newly-added mounts or extension-code edits. The `odrive-locked` emblem is installed but not yet applied — wired up when we have a way to detect upstream-locked files.

**Double-click intercept** is handled at the FreeDesktop MIME layer rather than via a Nautilus plugin (Nautilus 4 doesn't expose a Python activation hook). `odrive-cli install-handlers` writes `~/.local/share/mime/packages/odrive-linux.xml`, `~/.local/share/applications/odrive-linux-open.desktop`, and a hicolor icon set under `~/.local/share/icons/hicolor/<size>/{emblems,mimetypes}/`. The MIME XML registers `application/vnd.odrive.placeholder-file` for `*.cloud` and `application/vnd.odrive.placeholder-folder` for `*.cloudf` — each bound to a generic `odrive-cloud-file` / `odrive-cloud-folder` icon installed from `odrive-icons/mime_icons/{cloud-file,cloud-folder}/` so a plain `*.cloud` / `*.cloudf` renders distinctly in Nautilus — plus seven cloud-file-type sub-MIMEs (`application/vnd.odrive.{gdoc,gsheet,gslides,gdraw,gform,gmap,onenote}-cloud`) that match `.gdocx.cloud` etc., are `<sub-class-of>` placeholder-file (so the .desktop handler still applies), and bind to vendor-prefixed icon names via `<icon name="odrive-<type>-cloud"/>` — sub-type icons take precedence over the generic parent icon for files matching their globs. The .desktop calls `odrive-cli open %f`. `odrive-cli open <path>` runs `agent.sync()` on the placeholder, strips the `.cloud`/`.cloudf` suffix, and `xdg-open`s the materialized result. Icons are sourced from the workspace `odrive-icons/` directory discovered relative to the running binary — if absent (e.g. installed without copying assets), MIME registration still happens and Nautilus falls back to generic icons. `uninstall-handlers` reverses everything (files + cache refresh) including sweeping our installed icon files. **For deb/rpm packaging see `memory/project_packaging_split.md`** — the icons / MIME / desktop bits move to package files; only the placeholder-padding stays as a runtime command.

**Mount-root icon.** `odrive-cli install-handlers` installs `odrive-icons/mime_icons/main-folder/main-folder-512x512.png` into hicolor's `places/` category as `odrive-mount-folder` (sizes: `512x512`, `256x256`, `48x48` — single source PNG copied to each so Nautilus's render-time scaler can pick a sensible bucket). Per-mount selection is via GVFS metadata: `OdriveAgent::mount` calls `odrive_core::set_folder_custom_icon(local, "odrive-mount-folder")` after a successful `odrive mount`, which shells out to `gio set <local> metadata::custom-icon-name odrive-mount-folder`. `OdriveAgent::unmount` reverses it via `unset_folder_custom_icon`. `install-handlers` also walks every existing mount and applies the icon, so users who already had mounts before the feature landed get the icon back-filled at install time. Both helpers are best-effort — a non-GVFS environment or a missing `gio` binary just leaves the default folder icon. Restart Nautilus (`nautilus -q`) to pick up changes; the metadata is read on file-list refresh.

**Why placeholders are padded to one byte.** GLib's `g_content_type_guess` hardcodes empty files to `application/x-zerosize` *before* checking glob patterns, so a zero-byte `.cloud` would never resolve to our MIME type and double-click would fall through to the generic empty-file handler. The upstream odrive agent identifies placeholders by the `.cloud`/`.cloudf` extension, not by zero size — confirmed by test — so a single null byte is invisible to it but lets our glob match win for content-type resolution. `odrive-core::pad_placeholder` is the shared implementation; `scan_placeholders` calls it on every placeholder it encounters, the Nautilus extension's `update_file_info` calls it opportunistically as Nautilus enumerates directories, and `install-handlers` runs a one-shot scan over every mount immediately after registering the MIME types.

## Non-obvious things to know before editing

- **Mount enumeration goes through `odrive status --mounts`, not `odrive mounts`.** The upstream CLI has no `mounts` subcommand — that name is *only* the `odrive-cli` wrapper subcommand we expose. The agent prints two lines per mount (`<local>  status:<state>` then `<remote>  status:<state>`, with remote rendering blank for the odrive root `/`); `parse_mounts` handles that pairing.
- **`is_running` requires both a live agent process *and* a clean `odrive status` exit.** Process aliveness comes from `pgrep -f <agent_path>` (stable contract regardless of upstream wording); the status exit catches the brief window where the process is up but the IPC isn't yet bound or has wedged. `get_status` shares the same process check so the two never disagree.
- **`scan_placeholders` is fault-tolerant per-entry.** Unreadable directory entries, recursion errors, and DB upsert failures all `log::warn!` and continue rather than aborting the scan. The returned count only includes successfully-recorded placeholders.
- **The GUI's `update_ui` closure must stay `Clone` and `'static`-friendly.** It's cloned into each button handler. Adding non-`Clone` captures will break the build in non-obvious ways.
- **Long-running agent calls go through `worker::spawn`.** `odrive-gui/src/worker.rs` is a tiny wrapper that runs the work on `std::thread::spawn` and polls a `mpsc::sync_channel` from the GTK main loop via `glib::timeout_add_local` (100ms). Quick CLI calls (status, mount, authenticate, threshold setters, foldersyncrule) stay on the main thread; the slow ones (`install_official`, `sync_recursive`, `scan_placeholders`) use the worker so the UI keeps painting. `OdriveAgent` derives `Clone` specifically so a fresh copy can move into the worker thread (its fields are just owned `String`s, no shared state).
- **`odrive-core` re-exports `OdriveDb` from `lib.rs`.** Use `odrive_core::OdriveDb`, not `odrive_core::db::OdriveDb`.
- **Tray icons must land under `48x48/status/` even though our masters are 256+px.** SNI hosts on GNOME (the `status-tray` extension and the older `appindicator` one) walk a hardcoded list of panel-relevant size buckets — `48x48`, `32x32`, `24x24`, `22x22`, `16x16` — when resolving an `IconName` against the icon theme. They never check `256x256+/`. `install_tray_icon` deposits the smallest available source PNG (256-px) under `48x48/status/<name>.png` *in addition to* the per-size hi-res copies; dimensions don't have to match the bucket name (gtk-update-icon-cache and St.Icon both load by file path and scale at render time), but the file has to exist somewhere the host walks or the icon goes blank.
- **Tray animation frames live under `odrive-icons/tray-icons/animated/<color>/` with per-colour filename quirks.** Pink frames are `infinity-1.png`…`infinity-16.png`; black/white frames are `infinity-<color>-backup-1.png`…`-16.png`. `install_tray_animation` ignores the prefix and parses only the trailing `-N.png` to derive the frame number, then installs each as `odrive-tray-<color>-active-<N>` under both `256x256/status/` and `48x48/status/` (same panel-host shim story as the static icons). Colours without an `animated/<color>/` directory (`darkgrey`, `grey`) silently install zero frames; the GUI detects this at runtime via `render_frames_for_color` returning `None` and falls back to the default colour's frames (pink) — the idle icon still respects the user's pick, only the active-state animation borrows pink. Frame count is hardcoded at 16 in both crates — change `TRAY_ANIMATION_FRAMES` in `odrive-cli/src/main.rs` *and* `ANIMATION_FRAMES` in `odrive-gui/src/indicator.rs` together if the asset bundle's count ever changes.

## Reference: the upstream `odrive` CLI surface this code wraps

```
odrive status                # overview (also prints a Mounts: N count)
odrive status --mounts       # list mounts (two lines each: local then remote)
odrive mount <local> <remote>
odrive unmount <local>
odrive sync <path>           # download a .cloud, or expand a .cloudf
odrive sync <path> --recursive --nodownload   # placeholder-only expansion (typical first run after mount)
odrive unsync <path>         # revert local file to .cloud placeholder
odrive unsync <path> --force # also discard un-uploaded local changes
odrive refresh <path>        # re-check remote for changes
odrive authenticate <key>
odrive placeholderthreshold <never|small|medium|large|always>   # auto-download files under threshold size on expand
odrive xlthreshold <never|small|medium|large|xlarge>            # split files larger than this into chunks on upload
odrive autounsyncthreshold <never|day|week|month>               # auto-cleanup files not accessed within window (premium)
odrive foldersyncrule [--expandsubfolders] <path> <0|inf|N>     # per-folder auto-download rule; no list/remove command exists
odrive shutdown                                                 # terminate the agent cleanly
```

**No upstream LIST or REMOVE for foldersyncrule.** The agent only accepts the setter form. To "remove" a rule we set the threshold to `0` (= never auto-download) and drop our SQLite row. To "list" rules we read our SQLite row directly — what the user set via the Manager. Rules set via the upstream CLI directly are invisible to us.

**Threshold-token asymmetry to know about.** The CLI accepts `never`/`always` for `placeholderthreshold` and `xlarge` for `xlthreshold`; the *same* values render in `odrive status` text as `neverDownload` / `alwaysDownload` / `extraLarge`. `odrive-core::parse_global_settings` accepts both renderings; the CLI-arg side uses the short form via `<Enum>::as_cli_arg()`.
